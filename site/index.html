Why intent?

Every generation of AI tooling reduced the gap between what you mean and what the AI builds. Intent closes it.

Era 1 — Prompt
"Build me a password reset flow."
AI fills in the blanks. Behavior varies with model, phrasing, and temperature. Hallucinations creep in wherever the spec is silent.
→
Era 2 — Context
"Here are my requirements doc, schema, and existing code. Now build the reset flow."
AI infers from context. More consistent — but it still interprets. Different runs produce different results. No way to verify alignment.
→
Era 3 — Intent
CONTRACT Reset {
  ENSURES { "Link is sent" }
  VERIFY  { "Single-use only" }
}
AI verifies against a contract. Explicit invariants leave no room to guess. Same spec → same output across agents, models, and sessions.

Inference vs. contract

When AI has no contract, it guesses what "correct" means. A .intent file defines it — so the agent can verify its own output.

AI infers — no contract
"Users reset their password by email. The link should expire. Tokens are single-use. Log resets for auditing..."

AI decides what "correct" means. Every run interprets the spec differently. No way to verify alignment after the fact.

AI verifies — explicit contract
INTENT PasswordReset {
  SUMMARY: "Email-based reset flow."
  REQUIREMENTS {
    - "Token expires after 24h."
    - "Single-use only."
    - "Log all resets."
  }
  CONTRACT {
    ENSURES { "Reset link is sent" }
    VERIFY  { "Single-use enforced" }
  }
}

"Correct" is defined. AI checks its output against the contract. Same spec → same behavior across every agent and session.

Get started in 4 steps

Each step uses one specialized agent. Paste the prompt into Claude, Cursor, Copilot, or any chat-based assistant and follow its guidance. No new tools required.

1
@aim-author Start here

Describe your feature, get a structured spec

Tell the Author agent what you want to build in plain English — the actors, rules, and edge cases. It asks clarifying questions, then generates a .intent file and saves it to your repo. This file is the contract everything else builds from.

2
@aim-implementer

Point the Implementer at your intent file

The Implementer reads the .intent file and synthesizes production-ready code and tests. Working from an explicit contract — not a vague prompt — it knows what to build, what the edge cases are, and what "done" looks like.

3
@aim-verifier

Verify alignment before you merge

Run the Verifier to compare your implementation against the intent. It produces a drift report: which requirements are met, which CONTRACT invariants are satisfied, and which — if any — have been broken or missed.

4
@aim-repairer When needed

Repair drift before it becomes a bug

If the Verifier finds drift, the Repairer fixes it — updating the code to match the intent, or updating the intent to reflect a deliberate change. The branch lands clean, with code and spec in sync.

Verifier

Checks that your code still matches the intent after changes. Use this before merging to catch drift before it becomes a bug.

Once ready, confirm you have successfully parsed the v2.2 specification and mapped the relationship between intent and implementation. Do not produce a drift report yet.
Repairer

Restores alignment when code and intent diverge — fixes the code to match the spec, or updates the spec to reflect what changed.
