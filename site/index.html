Intent-First Programming

Not another spec.md. AIM shifts AI software work from passing around context to operating on intent: structured, versioned artifacts agents can build from, verify against, and repair over time.

Beyond spec.md

A markdown file gives an agent context. AIM gives it explicit behaviors, contracts, and invariants that can survive implementation, verification, and repair.

Password Reset

From a sentence to enforceable behavior

spec.md
"Users can reset their password by email."
AIM
INTENT PasswordReset { ACTOR User ACTION RequestReset REQUIRES { "User owns the email address" } ENSURES { "Reset link is sent" } VERIFY { "Expired tokens cannot be reused" } }

Agents can build the flow and later verify ownership, token expiry, and single-use behavior.

Team Invites

From feature request to operating contract

spec.md
"Admins can invite members to a workspace."
AIM
INTENT TeamInvites { ACTOR Admin ACTION InviteMember ENSURES { - "Invite includes a role" - "Invite expires" - "Audit event is recorded" } }

Agents now have durable rules around authorization, role assignment, expiry, and auditability.

One Clean Change at a Time

AIM works best as a commit-scoped workflow: start clean, author intent, synthesize code, align drift, then commit and push. Each run stays focused on one part of the system.

0

Start from a clean repo

Begin from a committed baseline so the agent sees one stable source of truth and one clear diff.

1

Run an intent agent

Use an authoring agent to write or refine the AIM files first, so product meaning is explicit before implementation begins.

2

Run a coding agent

Hand that intent layer to an implementation agent that synthesizes code from explicit intent instead of guessing from free-form context.

3

Run alignment and repair

Use verifier and repair passes to compare the code with the intent and restore alignment before the change leaves the branch.

4

Commit and push

Ship one focused change. That commit becomes a clean handoff point for a human reviewer or for another agent to pick up later.

Lower context load for agents Each run only needs the slice of intent, code, and diff related to one change, which keeps the context window focused instead of flooding it with the whole system.
Commit-scoped handoff After you commit, a different agent can pull that exact state and continue the next step without replaying the whole chat history or re-learning the entire repo.
Better iteration over time Intent, implementation, and repair stay separable, which makes it easier to revisit a branch later and run another pass with a better model or a different specialized agent.

How AIM Runs in Practice

A structured workflow with four agent roles. Teams can use the whole loop, or skip verification and repair until drift actually matters.

1) Intent Agent

Works with the developer to define and refine the intent layer, without jumping ahead into implementation details.

Initialize as an AIM v2.2 Agent (https://intentmodel.dev/brain.md) in the Intent Author role. I want to describe a new component.

2) Coding Agent

Synthesizes working code from the intent layer, keeping implementation aligned with the product model.

Initialize as an AIM v2.2 Agent (https://intentmodel.dev/brain.md) in the Implementer role. I want to build a package.

3) Verifier Agent

Checks implementation against intent, produces a drift report, and highlights when the source of truth needs to change.

Initialize as an AIM v2.2 Agent (https://intentmodel.dev/brain.md) in the Verifier role. Give me a drift report.

4) Repair Agent

Restores alignment by fixing implementation or revising intent after the verifier has identified the drift.

Initialize as an AIM v2.2 Agent (https://intentmodel.dev/brain.md) in the Repairer role. Restore alignment.
