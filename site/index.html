Get Started in 3 Steps

AIM works with whatever AI coding assistant you already use — Claude, Cursor, Copilot, or any chat-based agent. No new tools required to try it.

1

Describe what you want to build

Pick a component or feature. It can be something new ("a password reset flow") or something you want to verify already exists ("our team invite system"). You don't need any AIM files yet — just a plain description.

2

Paste the Author prompt into your AI assistant

Copy the Intent Author prompt from the Agent Roles section below and paste it into your coding assistant. The agent will ask questions and generate a .intent file — a structured contract for the feature.

3

Hand the intent file to a coding agent

Copy the Implementer prompt, point it at the .intent file the author just created, and let it synthesize the code. From here you can also verify and repair using the other roles.

Or skip straight to a working example

Pull a ready-made package from the registry and have it built in your stack in one prompt. Great for seeing the full loop before authoring your own intent.

Beyond spec.md

A markdown file gives an agent context. AIM gives it explicit behaviors, contracts, and invariants that can survive implementation, verification, and repair.

Password Reset

From a sentence to enforceable behavior

spec.md
"Users can reset their password by email."
AIM
AIM: auth.reset#intent@2.2 INTENT PasswordReset { SUMMARY: "Email-based reset flow." REQUIREMENTS { - "User can request reset." } CONTRACT Request { SUMMARY: "Rules" ENSURES { "Reset link is sent" } VERIFY { "Tokens expire after use" } } }

Agents can build the flow and later verify ownership, token expiry, and single-use behavior.

AI Agent Roles

AIM defines four specialized roles for AI agents. Paste these prompts into your coding assistant to initialize them into the correct context.

Role 1
Start here

Intent Author Agent

Turns requirements into structured AIM files. Use this for new features or spec refinements.

Initialize as an AIM v2.2 Agent by reading the instructions at https://intentmodel.dev/brain.md. Operating Role: Intent Author role. I want to describe the intended behavior for [Component Name]. Help me draft the .intent envelope and necessary facets (Schema, Contract, Flow). Ask me about the core requirements first.
Role 2

Implementer Agent

Synthesizes code and tests directly from AIM source files. Use this to build the feature.

Initialize as an AIM v2.2 Agent by reading the instructions at https://intentmodel.dev/brain.md. Operating Role: Implementer role. Command: build [Package Name] in [Tech Stack]. Execute the full protocol: 1. Fetch and materialize the package into local ./aim/ using the V2.2 Nested Layout. 2. Synthesize the production-ready code and tests.
Role 3

Verifier Agent

Compares implementation against intent to detect drift. Use this to ensure quality before landing a change.

Initialize as an AIM v2.2 Agent by reading the instructions at https://intentmodel.dev/brain.md. Operating Role: Verifier role. I want to verify my local implementation against the intent files in ./aim/[Package Name]. Compare the logic, schemas, and contracts. Output a DRIFT REPORT highlighting any mismatches.
Role 4

Repairer Agent

Restores alignment when code and intent diverge. Use this to fix bugs or update specs.

Initialize as an AIM v2.2 Agent by reading the instructions at https://intentmodel.dev/brain.md. Operating Role: Repairer role. My intent and implementation are out of sync for [Package Name]. Read the drift and restore alignment. Prefer fixing the code unless I specify the intent is outdated.

One Clean Change at a Time

AIM works as a 4-step loop. Each step shows what you do, what AIM does, and what moves forward in the repo.

You do

Describe the feature in plain language

Tell the authoring agent the actors, actions, rules, and invariants for the component. Stay at the product level; do not describe implementation yet.

AIM does

Writes the intent layer

The Author agent turns that description into versioned .intent files with explicit contracts, behaviors, and verification rules the rest of the loop can rely on.

Initialize as an AIM v2.2 Agent by reading the instructions at https://intentmodel.dev/brain.md. Operating Role: Intent Author role. I want to describe a new component.
Result

The repo gains a durable source of truth

The change is now grounded in structured intent instead of scattered chat context, so the coding agent can build from something explicit and reusable.

You do

Run the coding agent on /aim

Point the coding agent at the intent layer and ask it to implement the feature from those files. The meaning already lives in the repo.

AIM does

Synthesizes code from explicit intent

The coding agent maps functions, types, UI, and side effects back to concrete intent blocks instead of improvising from a free-form prompt.

Initialize as an AIM v2.2 Agent by reading the instructions at https://intentmodel.dev/brain.md. Operating Role: Implementer role. I want to build a package.
Result

The repo gains working implementation

You now have code and intent moving together as one change, with a traceable path back to the contracts that defined the work.

You do

Run alignment before the change lands

Ask AIM to verify the implementation against the intent and, if drift exists, run repair before the change leaves the branch.

AIM does

Detects drift and restores alignment

The verifier compares code and intent, produces a drift report, and the repair pass updates the right layer so the source of truth and implementation match again.

Initialize as an AIM v2.2 Agent by reading the instructions at https://intentmodel.dev/brain.md. Operating Role: Verifier role. Give me a drift report.
Initialize as an AIM v2.2 Agent by reading the instructions at https://intentmodel.dev/brain.md. Operating Role: Repairer role. My intent and implementation have diverged. Help me restore alignment.
Result

Intent and code are back in sync

The branch is ready to hand off without hidden mismatches between what the feature means and what it actually does.

You do

Commit and push the change

Land the intent files, implementation, and alignment updates together as one focused commit.

AIM does

Stops at a clean handoff point

No additional synthesis is needed. The loop leaves the branch in a stable state another agent or reviewer can pick up without replaying the whole session.

Result

One reviewable, reusable change

Because the loop stayed narrow, the next agent can pull this commit, load only the relevant context, and continue iterating without dragging the whole repo into the prompt.

Lower context load for agents Each run only needs the slice of intent, code, and diff related to one change instead of the whole system.
Clean handoff between runs Each commit becomes a stable checkpoint a different agent or reviewer can pick up later without reconstructing the entire conversation.
Better long-term iteration You can revisit the branch later, pull the exact state, and run another pass with a different model or another specialized agent.

Built for scale.

  • Shared Central Truth The /aim/ folder is checked into your repo as the authoritative reference. It is the shared "brain" that coordinates all agents working on the project.
  • Model Specialization Use reasoning-focused models for authoring and verification, and coding-focused models for implementation.
  • Native Package Registry Fetch and materialize packages into namespaced nested paths (/aim/game/snake/) automatically using registry-native agents.
